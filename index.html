KPI Grid / Dashboard Update
Application Architecture Mapped — Codebase, Schema & Query Logic
Carried out a full architectural review of the FATP dashboard application — going through the codebase module by module, mapping the business logic layer, and documenting the underlying database schema along with table relationships and foreign-key dependencies. Each query used across the reporting layer was traced back to its source table and cross-checked against the fields actually consumed on the frontend.

This groundwork is meant to close the knowledge gap before the next phase of development — anyone picking up the codebase going forward should be able to follow the data flow from raw MES tables through to the dashboard views without having to reverse-engineer it from scratch. The documentation also surfaced a few redundant joins and a couple of tables that were being queried but never actually rendered, which will be cleaned up in the next pass.
Documented
Architecture Mapping
MES Database Connectivity Validated — Queries Optimized & Rewritten
Tested the MES database connection end to end and reviewed the integration requirements against what the dashboard actually needs to pull. A number of the original queries were rewritten — some were doing unnecessary full table scans where an indexed lookup would do, and a few joins were restructured to avoid pulling more rows than the aggregation step actually needed.

Net result is lower query execution time and a connection layer that's been validated under repeated test runs rather than just a single successful connect. Integration requirements between the MES schema and the dashboard's data model have also been documented so future schema changes on the MES side can be flagged early.
In Progress
Code-Level Debugging & Impact Analysis
Went through code-level debugging and dependency tracing to identify where individual modules touch shared logic — particularly around the data transformation layer, where a change in one query could silently affect three or four downstream views. Impact analysis was done before pushing any fix, so each change could be verified against the modules it actually touches rather than assuming isolation.

This has helped catch a couple of issues early, including a stale cache reference and a mismatched data type between two tables that was causing silent truncation on one of the numeric fields.
Troubleshooting

Response Time Cut from ~5s to ~2s — Backup & Monitoring Established
Spent time on overall performance tuning for the system, bringing average response time down from roughly 5 seconds to about 2 seconds. Most of the gain came from trimming redundant database round-trips on the lookup endpoint and adding indexes on the fields used most often in the filter queries.

Alongside that, database backup, monitoring, and maintenance procedures have been put in place — scheduled backups are now running, and basic health checks are in place to catch issues before they affect uptime. This should protect data integrity going forward and give a recovery path if something does go wrong.

ESP32 Project for Temperature and Humidity Sensing
End-to-End Pipeline Completed
The full sensing pipeline is now complete and running: ESP32 → DHT22 → Arduino → HiveMQ → MongoDB → Dashboard. The ESP32 reads temperature and humidity off the DHT22 over a single-wire digital interface, the Arduino layer handles the serial bridging and packages each reading into a payload, and that payload is published to a HiveMQ broker over MQTT. From there, a subscriber service picks up the message and writes it into MongoDB as a time-stamped document, which the dashboard then queries for display.

Getting this chain stable took a few iterations — the original setup had occasional packet drops on the MQTT publish step under bad WiFi conditions, so a basic retry/QoS adjustment was added on the publish side, and the Mongo writes were moved to a small buffered queue to avoid losing readings if the connection dropped for a few seconds.
Pipeline Live

The second is a calendar-based lookup, letting anyone pull up the recorded sensor data for any specific date stored in MongoDB, rather than only seeing the live feed.
Planned
Feeder List Generator
FeedSync Amended — QPS & Alternate Partcode Handling
FeedSync has been amended to align with new requirements around QPS and alternate partcode handling. Previously, a change in alternate partcode consumption would trigger generation of a fresh feeder list file each time, which meant the SMT team was dealing with multiple file versions for what was effectively the same setup with a substituted part code.

The logic has been reworked so that alternate partcode consumption is handled within a single feeder list generation pass instead of forcing a new file, cutting down on the duplicate-file problem and making it clearer which file is the current source of truth during line changeover.
Amended
